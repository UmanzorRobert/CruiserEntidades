Plan de Fases de CruiserCat

BLOQUE 1 --- FUNDAMENTOS

 Fase 1:  Setup solución & estructura de capas
-  1.1.  Crear solución .NET 10 con proyectos: Entities, DataAccess, Business, Web.
-  1.2.  Configurar referencias entre capas y convención de nombres (namespace Cruiser.\*).
-  1.3.  Agregar NuGet base: EF Core 10, Identity, BCrypt.Net, FluentValidation.
-  1.4.  Setup Cmder + PowerShell integrado en VS 2026.
-  1.5.  Configurar .editorconfig, .gitignore y estructura inicial de carpetas por módulo.
-  1.6.  Primer commit a GitHub con estructura base y pipeline CI esqueleto.

 Fase 2:  Docker + PostgreSQL 18 + entorno local
-  2.1.  Crear docker-compose.yml con postgres:18 y schemas pre-creados.
-  2.2.  Configurar volúmenes, variables de entorno y red Docker interna.
-  2.3.  Verificar conexión desde VS con pgAdmin 4 y DBeaver.
-  2.4.  Configurar appsettings.json con connection string base + search_path por contexto.
-  2.5.  Setup de múltiples entornos: Development / Staging / Production con secrets separados.
-  2.6.  Documentar proceso de arranque local en README.md con diagrama de arquitectura.

 Fase 2B:  Secrets & gestión de entornos
-  2B.1.  Configurar User Secrets para desarrollo local.
-  2B.2.  Crear script init-dev-secrets.ps1 para provisionar secrets.
-  2B.3.  Definir convención de variables de entorno por contexto: DB\_\_\[Contexto\]\_\_ConnectionString.
-  2B.4.  Configurar Railway Secrets para staging/producción.
-  2B.5.  Crear appsettings.Development.json y .Staging.json (solo estructura).
-  2B.6.  Documentar variables requeridas en SECRETS.md.

 Fase 3:  EntidadBase + Fluent API Base
-  3.1.  Implementar EntidadBase con Id (Guid), FechaCreacion, FechaModificacion, EstaActivo.
-  3.2.  Crear IEntityTypeConfiguration\<T\> base con soft-delete filter y conversores UTC.
-  3.3.  Crear CruiserIdentityDbContext heredando de IdentityDbContext\<ApplicationUser\> (schema "identity").
-  3.4.  Implementar OnModelCreating con llamadas a configuraciones por módulo.
-  3.5.  Configurar conversores UTC para todas las DateTime.
-  3.6.  Definir convención de naming para tablas: snake_case con HasDefaultSchema.

 Fase 3B:  Implementar los 9 DbContexts de dominio
-  3B.1.  CruiserCoreDbContext (schema "core"): Config, Módulos, Permisos, Auditoría, Notificaciones, GDPR.
-  3B.2.  CruiserComercialDbContext ("comercial"): Cliente, Empleado, Cotización, Asignaciones, RFM.
-  3B.3.  CruiserInventarioDbContext ("inventario"): Producto, Almacén, Movimientos, Lotes, Fichas.
-  3B.4.  CruiserComprasDbContext ("compras"): Proveedor, OrdenCompra, Recepción, Discrepancias.
-  3B.5.  CruiserOperacionesDbContext ("operaciones"): TipoServicio, Contrato, OS, Checklist, Calidad.
-  3B.6.  CruiserFacturacionDbContext ("facturacion"): Factura, Serie, Pago, CuentaCobrar, VeriFactu.
-  3B.7.  CruiserRRHHDbContext ("rrhh"): ContratoLaboral, Fichaje, Ausencias, EPI, Nómina.
-  3B.8.  CruiserFlotaDbContext ("flota"): Vehículo, Mantenimiento, Ruta, Incidencia.
-  3B.9.  CruiserAnaliticaDbContext ("analitica"): MétricaKPI, HistorialKPI, DashboardUsuario.
-  3B.10.  Registrar todos los contextos en Program.cs con DI y connection string por contexto.

 Fase 4:  Migraciones iniciales por contexto --- DDL
-  4.1.  Migración inicial Identity: tablas Identity + ApplicationUser extendido.
-  4.2.  Migración inicial Core: tablas de configuración, módulos, permisos, sesiones.
-  4.3.  Migraciones iniciales para los 8 contextos de dominio restantes.
-  4.4.  Configurar HasColumnType("jsonb") en todos los campos JSON.
-  4.5.  Verificar generación correcta de tablas y schemas en PostgreSQL.
-  4.6.  Crear script apply-all-migrations.ps1 para aplicar en orden.

 Fase 4B:  Migraciones de índices --- GIN, compuestos y full-text
-  4B.1.  Migración Inv_Indexes: índices GIN en JSONB.
-  4B.2.  Migración Core_Indexes: índices en BitácoraAcción.
-  4B.3.  Migración Facturacion_Indexes: índices en Factura.
-  4B.4.  Migración RRHH_Indexes: índices compuestos en Fichaje.
-  4B.5.  Migración Operaciones_Indexes: índices en OrdenServicio.
-  4B.6.  Migración Analitica_Indexes: índice único en HistorialKPI.
-  4B.7.  Crear script validate-indexes.sql para verificar índices críticos.

 Fase 4C:  Outbox Pattern + cross-context consistency
-  4C.1.  Crear tabla OutboxMessage en Core: Id, ContextOrigen, TipoEvento, PayloadJson, Procesado.
-  4C.2.  Implementar IOutboxService que persiste eventos en la misma transacción.
-  4C.3.  Job Hangfire "OutboxProcessor" (cada 30s) para despachar mensajes.
-  4C.4.  Implementar IOutboxConsumer\<T\> para eventos cross-context.
-  4C.5.  Garantizar at-least-once delivery con reintentos exponenciales.
-  4C.6.  Dashboard de mensajes Outbox en administración.

 Fase 4D:  Migration CI/CD --- efbundle & pipeline
-  4D.1.  Configurar efbundle en GitHub Actions.
-  4D.2.  Step "Generate Migration Bundle": dotnet ef migrations bundle para cada contexto.
-  4D.3.  Step "Apply Migrations" en deploy a staging.
-  4D.4.  Rollback automatizado si el bundle falla.
-  4D.5.  Crear migración de smoke test: tabla MigrationHealthCheck.
-  4D.6.  Documentar proceso en MIGRATIONS.md.

 Fase 4E:  DbContext Health Checks & Connection Pooling
-  4E.1.  Registrar health check por DbContext en /health.
-  4E.2.  Configurar Npgsql connection pooling (MinPoolSize=2, MaxPoolSize=20).
-  4E.3.  Implementar AddDbContextPool\<T\> para contextos de alta carga.
-  4E.4.  Endpoint /health/db con latencia de ping.
-  4E.5.  Configurar retry policy en Npgsql.
-  4E.6.  Dashboard de conexiones activas por contexto.

 Fase 4F:  Testing de migraciones & seed testing
-  4F.1.  Proyecto CruiserCat.Tests.Migrations con xUnit + Testcontainers.
-  4F.2.  Test CanApplyAllMigrations.
-  4F.3.  Test CanRollbackLastMigration.
-  4F.4.  Test SeedDataIsIdempotent.
-  4F.5.  Test SchemaIsolation.
-  4F.6.  Integrar tests de migración en pipeline de CI.

Arquitectura de Múltiples DbContext
El sistema utiliza 10 DbContexts independientes, cada uno responsable de un dominio funcional. Todos comparten la misma base de datos PostgreSQL pero están aislados mediante schemas propios. Las relaciones entre dominios se resuelven por Guid (sin FK de base de datos), y la consistencia se garantiza en la capa de negocio (Business layer).
DbContext	Módulos / Entidades	Schema PG	Historia de migraciones
CruiserIdentityDbContext	IdentityUser, IdentityRole, Claims, Tokens, Sesion, AuditoriaLogin	identity	__MigrationsHistory_Identity
CruiserCoreDbContext	Config, Módulos, Permisos, Sidebar, Notificaciones, BitácoraAcción, AuditoriaEntidad, GDPR, ConsentGDPR, Solicitudes ARCO, Anonimización	core	__MigrationsHistory_Core
CruiserComercialDbContext	Cliente, Empleado, Cotización, LíneaCotización, AsignacionClienteEmpleado, SegmentaciónRFM	comercial	__MigrationsHistory_Comercial
CruiserInventarioDbContext	Producto, Categoría, UnidadMedida, Almacén, Ubicación, MovimientoInventario, InventarioFísico, Lote, FichaSeguridad	inventario	__MigrationsHistory_Inventario
CruiserComprasDbContext	Proveedor, ContratoProveedor, OrdenCompra, LíneaOC, RecepciónMercancía, DiscrepanciaRecepción	compras	__MigrationsHistory_Compras
CruiserOperacionesDbContext	TipoServicio, ZonaServicio, ContratoServicio, OrdenServicio, PlantillaChecklist, ChecklistOS, InspecciónCalidad, NPS	operaciones	__MigrationsHistory_Operaciones
CruiserFacturacionDbContext	Factura, LíneaFactura, SerieFactura, PagoFactura, CuentaCobrar, VeriFactu, ListaPrecio	facturacion	__MigrationsHistory_Facturacion
CruiserRRHHDbContext	ContratoLaboral, Fichaje, SolicitudAusencia, EPI, EntregaEPI, Formación, Nómina, LiquidaciónComisión, CategoríaEmpleado	rrhh	__MigrationsHistory_RRHH
CruiserFlotaDbContext	VehículoEmpresa, MantenimientoVehiculo, RutaVehiculo, IncidenciaVehiculo	flota	__MigrationsHistory_Flota
CruiserAnaliticaDbContext	MétricaKPI, HistorialKPI, DashboardUsuario, RegistroIntegración	analitica	__MigrationsHistory_Analitica

Las FK cross-context son Guid sin restricción de clave foránea en la BD. EF Core no navegará entre contextos; la Business layer resuelve las consultas cross-context con joins en memoria o vistas PostgreSQL.
Cada DbContext se registra en DI con AddDbContext<TContext> y su propia connection string con search_path al schema correspondiente. Esto permite que las migraciones Add-Migration siempre sepan a qué contexto pertenecen

Principios de Estrategia de Migraciones
Principio	Descripción
Una migración = Un cambio lógico	Nunca mezclar cambios de esquema con cambios de datos. Una migración para DDL (tablas/columnas), otra para DML (seeds/actualizaciones).
Naming convention	Prefijo con contexto: [ContextSuffix]_[YYYYMMDD]_[NNN]_[Descripción]. Ej: Inv_20260101_001_InitialInventario
Orden de aplicación	Identity → Core → Comercial → Inventario → Compras → Operaciones → Facturación → RRHH → Flota → Analítica
Schema isolation	Cada DbContext usa su propio PostgreSQL schema. Las FK cross-context usan Guid sin FK de base de datos (validación en capa de negocio).
Rollback documentado	Cada migración tiene script DOWN documentado en /docs/migrations/rollback/.
Idempotencia	Seeds en migraciones protegidos con IF NOT EXISTS. Nunca ejecutar seed si ya existen datos.
Migration bundles CI/CD	Generar migration bundle (efbundle) en pipeline GitHub Actions para deploy sin EF tools en producción.
Test de migración	Validar Apply + Rollback en base de datos de staging antes de mergear a main.
