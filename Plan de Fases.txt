Plan de Fases de CruiserSystem

Plan de Fases
BLOQUE 1 — FUNDAMENTOS
Fase 1: Setup solución & estructura de capas
1.1. Crear solución .NET 10 con proyectos: Entities, DataAccess, Business, Web
1.2. Configurar referencias entre capas y convenciones de nombres (namespace Cruiser.*)
1.3. Agregar paquetes NuGet base: EF Core 10, AspNetCore Identity, BCrypt.Net, FluentValidation
1.4. Setup Cmder + PowerShell integrado en Visual Studio Community 2026
1.5. Configurar .editorconfig, .gitignore y estructura inicial de carpetas por módulo
1.6. Primer commit a repositorio GitHub con estructura base y pipeline CI esqueleto

Fase 2: Docker + PostgreSQL 18 + entorno local
2.1. Crear docker-compose.yml con imagen oficial postgres:18 con schemas pre-creados
2.2. Configurar volúmenes, variables de entorno y red Docker interna
2.3. Verificar conexión desde Visual Studio con pgAdmin 4 y DBeaver como alternativa
2.4. Configurar appsettings.json con connection string base + search_path por contexto
2.5. Setup de múltiples entornos: Development / Staging / Production con secrets separados
2.6. Documentar proceso de arranque local en README.md con diagrama de arquitectura

Fase 2B: Secrets & gestión de entornos 
2B.1. Configurar User Secrets (dotnet user-secrets) para desarrollo local — NUNCA credenciales en appsettings
2B.2. Crear script PowerShell init-dev-secrets.ps1 para provisionar secrets de un desarrollador nuevo
2B.3. Definir convención de nombres de variables de entorno por contexto: DB__[Contexto]__ConnectionString
2B.4. Configurar Railway Secrets para staging y producción con variables de entorno segregadas
2B.5. Crear appsettings.Development.json, appsettings.Staging.json (sin datos sensibles, solo estructura)
2B.6. Documentar en SECRETS.md qué variables son requeridas y en qué entorno

Fase 3: EntidadBase + Fluent API Base
3.1. Implementar EntidadBase con Id (Guid), FechaCreacion, FechaModificacion, EstaActivo
3.2. Crear IEntityTypeConfiguration<T> base con soft-delete global query filter y conversores UTC
3.3. Crear CruiserIdentityDbContext heredando de IdentityDbContext<ApplicationUser> — schema "identity"
3.4. Implementar OnModelCreating en cada DbContext con llamadas a IEntityTypeConfiguration por módulo
3.5. Configurar conversores UTC para todas las propiedades DateTime en todos los contextos
3.6. Definir convención de naming para tablas: snake_case en schema correspondiente (EF HasDefaultSchema)

Fase 3B: Implementar los 9 DbContexts de dominio
3B.1. Crear CruiserCoreDbContext (schema "core"): Config, Módulos, Permisos, Auditoría, Notificaciones, GDPR
3B.2. Crear CruiserComercialDbContext (schema "comercial"): Cliente, Empleado, Cotización, Asignaciones, RFM
3B.3. Crear CruiserInventarioDbContext (schema "inventario"): Producto, Almacén, Movimientos, Lotes, Fichas
3B.4. Crear CruiserComprasDbContext (schema "compras"): Proveedor, OrdenCompra, Recepción, Discrepancias
3B.5. Crear CruiserOperacionesDbContext (schema "operaciones"): TipoServicio, Contrato, OS, Checklist, Calidad
3B.6. Crear CruiserFacturacionDbContext (schema "facturacion"): Factura, Serie, Pago, CuentaCobrar, VeriFactu
3B.7. Crear CruiserRRHHDbContext (schema "rrhh"): ContratoLaboral, Fichaje, Ausencias, EPI, Nómina
3B.8. Crear CruiserFlotaDbContext (schema "flota"): Vehículo, Mantenimiento, Ruta, Incidencia
3B.9. Crear CruiserAnaliticaDbContext (schema "analitica"): MétricaKPI, HistorialKPI, DashboardUsuario
3B.10. Registrar todos los contextos en Program.cs con DI y connection string por contexto

Fase 4: Migraciones iniciales por contexto — DDL
4.1. Migración inicial CruiserIdentityDbContext: tablas Identity estándar + ApplicationUser extendido
4.2. Migración inicial CruiserCoreDbContext: tablas de configuración, módulos, permisos, sesiones
4.3. Migraciones iniciales para los 8 contextos de dominio restantes (una migración por contexto)
4.4. Configurar HasColumnType("jsonb") en todos los campos JSON vía Fluent API en cada contexto
4.5. Verificar generación correcta de tablas y schemas en PostgreSQL 18 con pgAdmin
4.6. Crear script PowerShell apply-all-migrations.ps1 para aplicar todos los contextos en orden

Fase 4B: Migraciones de índices — GIN, compuestos y full-text
4B.1. Migración dedicada Inv_Indexes: índices GIN en JSONB (PermisosEspeciales, DatosAdicionales, PuntajeRFM)
4B.2. Migración dedicada Core_Indexes: índices en BitácoraAcción (UsuarioId, FechaCreacion, Módulo)
4B.3. Migración dedicada Facturacion_Indexes: índices en Factura (ClienteId, FechaEmision, Estado, NumeroSerie)
4B.4. Migración dedicada RRHH_Indexes: índices compuestos en Fichaje (EmpleadoId, FechaEntrada)
4B.5. Migración dedicada Operaciones_Indexes: índices en OrdenServicio (Estado, FechaProgramada, EmpleadoId)
4B.6. Migración dedicada Analitica_Indexes: índice único en HistorialKPI (MetricaKPIId, Periodo)
4B.7. Crear script validate-indexes.sql que verifica la existencia de todos los índices críticos

Fase 4C: Outbox Pattern + cross-context consistency
4C.1. Crear tabla OutboxMessage en CruiserCoreDbContext: Id, ContextOrigen, TipoEvento, PayloadJson, FechaCreacion, Procesado
4C.2. Implementar IOutboxService que persiste eventos en la misma transacción que el cambio de negocio
4C.3. Job Hangfire "OutboxProcessor" (cada 30s): lee mensajes no procesados y los despacha a sus consumidores
4C.4. Implementar IOutboxConsumer<T> para cada tipo de evento cross-context (ej: ClienteCreadoEvent → Facturación)
4C.5. Garantizar at-least-once delivery con reintentos exponenciales (máx 5 reintentos con Hangfire)
4C.6. Dashboard de mensajes Outbox en panel de administración: pendientes, fallidos, procesados hoy

Fase 4D: Migration CI/CD — efbundle & pipeline
4D.1. Configurar efbundle en el pipeline de GitHub Actions: genera ejecutable de migraciones sin EF tolos
4D.2. Step "Generate Migration Bundle" en pipeline: dotnet ef migrations bundle --self-contained para cada contexto
4D.3. Step "Apply Migrations" en deploy a staging: ejecutar bundle contra BD de staging con variables de entorno
4D.4. Rollback automatizado: si el bundle falla, ejecutar script de rollback desde /docs/migrations/rollback/
4D.5. Crear migración de smoke test: tabla MigrationHealthCheck con timestamp de aplicación por contexto
4D.6. Documentar en MIGRATIONS.md el proceso completo: crear, aplicar, revertir y resolver conflictos

Fase 4E: DbContext Health Checks & Connection Pooling
4E.1. Registrar health check por DbContext en /health: verifica conectividad de cada uno de los 10 contextos
4E.2. Configurar Npgsql connection pooling en cada contexto: MinPoolSize=2, MaxPoolSize=20 según módulo
4E.3. Implementar AddDbContextPool<T> para los 3 contextos de mayor carga: Operaciones, Comercial, Facturación
4E.4. Endpoint /health/db devuelve estado de cada contexto con latencia de ping a PostgreSQL
4E.5. Configurar retry policy en Npgsql: 3 reintentos con backoff exponencial en caso de timeout transitorio
4E.6. Dashboard de conexiones activas por contexto en panel de administración (Npgsql statistics)

Fase 4F: Testing de migraciones & seed testing
4F.1. Crear proyecto CruiserSystem.Tests.Migrations con xUnit + Testcontainers (PostgreSQL efímero)
4F.2. Test CanApplyAllMigrations: aplica todas las migraciones de todos los contextos en contenedor limpio
4F.3. Test CanRollbackLastMigration: verifica que dotnet ef database update [PreviousMigration] funciona por contexto
4F.4. Test SeedDataIsIdempotent: ejecuta seed dos veces y verifica que no hay registros duplicados
4F.5. Test SchemaIsolation: verifica que cada contexto solo accede a su propio schema PostgreSQL
4F.6. Integrar tests de migración como step obligatorio en el pipeline de CI (no merge sin test verde)

BLOQUE 2 — AUTENTICACIÓN & SEGURIDAD
Fase 5: ASP.NET Core Identity + gestión de usuarios
5.1. Configurar ApplicationUser extendido con campos propios: EmpleadoId, último acceso, avatar, tema UI
5.2. Setup IdentityRole + asignación de roles múltiples por usuario, registrado en CruiserIdentityDbContext
5.3. Registrar servicios Identity en Program.cs con opciones de contraseña fuertes y política de bloqueo
5.4. Crear UsuarioService + IUsuarioRepository con patrón Repository en DataAccess
5.5. FluentValidation para DTO de creación/edición de usuario con validación de NIF español
5.6. Seed de usuario SuperAdmin inicial con hash BCrypt almacenado en migración de CruiserCoreDbContext

Fase 6: JWT + Refresh Tokens + cookies HttpOnly
6.1. Generar JWT con claims: UserId, Email, Roles, PermisosEspeciales, SchemaActivo
6.2. Implementar RefreshToken almacenado en CruiserIdentityDbContext + cookie HttpOnly SameSite=Strict
6.3. Endpoint POST /auth/refresh para renovación silenciosa de token con validación de IP y UserAgent
6.4. Endpoint POST /auth/logout con revocación de refresh token y limpieza de cookie
6.5. Middleware de validación JWT en cada request protegido con extracción de claims al contexto HTTP
6.6. Test en Swagger: login → token → acceso a recurso protegido → refresh → logout

Fase 7: BCrypt, política contraseñas & auditoría auth
7.1. Configurar BCrypt.Net con work factor 12 para hashing de contraseñas
7.2. Política: mínimo 8 chars, mayúscula + minúscula + número + carácter especial
7.3. Bloqueo de cuenta tras 5 intentos fallidos (lockout 15 min, configurable en ConfiguracionSistema)
7.4. Registro de intentos fallidos en tabla AuditoriaLogin (CruiserIdentityDbContext)
7.5. Endpoint de cambio de contraseña con validación de contraseña antigua + historial de últimas 5
7.6. Flujo de recuperación de contraseña vía email con token de un solo uso (TTL 1h)

Fase 8: Middleware, Serilog, Rate Limiting & Health Checks
8.1. Configurar Serilog con sinks: Console + File (rolling daily) + PostgreSQL (tabla LogEntry en CruiserCoreDbContext)
8.2. Estructura de log enriquecida: RequestId, UserId, IP, Módulo, ContextoDbActivo, CorrelationId
8.3. Configurar AspNetCoreRateLimit: 100 req/min por IP global, 10/min en /auth/*
8.4. Middleware global de manejo de excepciones: captura, log Serilog, respuesta Problem Details (RFC 7807)
8.5. Health Check /health con verificación de los 10 DbContexts + SignalR + Hangfire
8.6. Configurar correlación de requests con header X-Correlation-Id propagado a logs y respuestas

Arquitectura de Múltiples DbContext
El sistema utiliza 10 DbContexts independientes, cada uno responsable de un dominio funcional. Todos comparten la misma base de datos PostgreSQL pero están aislados mediante schemas propios. Las relaciones entre dominios se resuelven por Guid (sin FK de base de datos), y la consistencia se garantiza en la capa de negocio (Business layer).

Las FK cross-context son Guid sin restricción de clave foránea en la BD. EF Core no navegará entre contextos; la Business layer resuelve las consultas cross-context con joins en memoria o vistas PostgreSQL.
Cada DbContext se registra en DI con AddDbContext<TContext> y su propia connection string con search_path al schema correspondiente. Esto permite que las migraciones Add-Migration siempre sepan a qué contexto pertenecen

Principios de Estrategia de Migraciones
Principio	Descripción
Una migración = Un cambio lógico	Nunca mezclar cambios de esquema con cambios de datos. Una migración para DDL (tablas/columnas), otra para DML (seeds/actualizaciones).
Naming convention	Prefijo con contexto: [ContextSuffix]_[YYYYMMDD]_[NNN]_[Descripción]. Ej: Inv_20260101_001_InitialInventario
Orden de aplicación	Identity → Core → Comercial → Inventario → Compras → Operaciones → Facturación → RRHH → Flota → Analítica
Schema isolation	Cada DbContext usa su propio PostgreSQL schema. Las FK cross-context usan Guid sin FK de base de datos (validación en capa de negocio).
Rollback documentado	Cada migración tiene script DOWN documentado en /docs/migrations/rollback/.
Idempotencia	Seeds en migraciones protegidos con IF NOT EXISTS. Nunca ejecutar seed si ya existen datos.
Migration bundles CI/CD	Generar migration bundle (efbundle) en pipeline GitHub Actions para deploy sin EF tools en producción.
Test de migración	Validar Apply + Rollback en base de datos de staging antes de mergear a main.
